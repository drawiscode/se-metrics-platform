目标：帮助团队**快速接手/优化/持续开发**已有项目，通过“数据度量 + AI 分析解读 + 看板闭环任务”提升工程质量与协作效率。

## 环境配置相关
(1)在backend目录下面创建一个config/config.env文件，在里面填写如下字段：
DEVINSIGHT_DB=你实际的data/devinsight.db文件所在绝对路径
PORT=8080
GITHUB_TOKEN=你从github上面获得的token
在CMakeLists.txt中第一行vcpkg路径改为你的实际vcpkg所在路径

(2)第三方库依赖下载:你需要通过vcpkg下载
sqlite3
nlohmann-json
cpp-httplib
openssl
四个依赖库到本机，同时根据你具体vcpkg的下载路径来更改CMakeLists.txt的路径设置，这里建议你直接把vcpkg下载到C盘，即C：,否则你可能需要更改CMakeLists的以下内容：
target_include_directories(devinsight_backend PRIVATE
    ${CMAKE_CURRENT_SOURCE_DIR}/src
    C:/vcpkg/installed/x64-windows/include 
)
同时考虑到直接下载得到的sqlite3貌似不支持mingw,建议你使用MSVC来进行编译。

(3)你的所有被编译的文件应该以UTF-8 with BOM的形式保存，否则你可能通不过MSVC的编译，会报很奇怪的错误，导致你链接不到别的头文件和第三方库

(4)你选择的工具包应该是amd64而不是x86(如果你完全按照我的环境配置)

(5) .vscode的settings.json里面选择你实际的CMake编译器

## 1. 项目概述
本平台自动采集软件项目的多源数据(GitHub 仓库协作、CI/CD、静态分析结果、运行日志)，形成**可量化指标体系**与**项目健康度**。在此基础上引入 AI 能力对数据进行解释、总结与建议输出，并通过“看板与任务”将问题转化为行动，形成闭环。

### 1.1 数据来源说明（边界）
- **GitHub API 可获取**:Commit/PR/Issue/Release/Contributor 等仓库协作数据，部分 CI 信息(GitHub Actions)。
- **运行日志不来自 GitHub**：日志与运行状态监控子系统的采集对象为**接入的业务服务/演示服务**（通过日志 SDK 主动上报，或 Agent 采集本地日志文件）。
- **CI/CD(可选)**:
  - GitHub Actions:REST API 获取 workflow runs
  - Jenkins:Webhook + API(若环境允许)

## 2. 核心功能（模块划分）
平台由四个子系统组成： 仓库协作度量、代码质量分析、日志与运行状态监控、 AI驱动的智能分析与决策。

# 2.1 仓库协作度量子系统(Repo Metrics)
目标：回答“项目是否活跃？协作是否健康？响应是否及时？发布节奏是否稳定？”

**功能:**

1. **仓库接入与管理**
   - 支持用户添加要监控的 GitHub 仓库（`owner/repo`）。
   - 为每个仓库配置：
     - 所属团队 / 项目
     - 是否自动同步（定时拉数据）
     - 同步周期（如每天、每小时）

2. **数据采集**
   - 使用 GitHub API 拉取：
     - Issues：状态、创建时间、关闭时间、标签、评论数等
     - Pull Requests：创建/合并时间、变更文件数、参与 reviewer
     - Commits：作者、时间、增删行数
     - Releases：版本号、日期
   - 支持“全量同步 + 增量更新”，避免每次把全部历史拉完。

3. **协作与活跃度指标**
   - 活跃度：
     - 最近 N 天的提交次数折线图
     - 活跃贡献者数量（最近 N 天有提交的人）
   - 响应速度：
     - Issue 从创建到第一次回应（评论）的平均时间
     - Issue 从创建到关闭的平均时间
   - 协作质量：
     - 有 review 的 PR 占比
     - PR 平均合并时间
   - 发布节奏：
     - Release 频率、平均间隔

4. **仓库健康度评分**
   - 综合上面指标，给每个仓库打一个 0–100 的分数：
     - 活跃度子分数
     - 响应速度子分数
     - 协作质量子分数
     - 发布节奏子分数
   - 给出简单结论：
     - “健康”“一般”“有风险”等。


### (1) 仓库接入与管理
- 添加/删除/启用/停用监控仓库(owner/repo)
- 配置同步策略：全量同步、增量同步、同步周期
- 支持按“项目”聚合多个仓库（对接 #2.4 项目管理）

### (2) GitHub 数据采集与同步(REST API)
同步对象（可按阶段逐步实现）：
- 仓库信息:stars/forks/open issues/default branch/更新时间
- Issues:open/closed、创建/关闭时间、评论数、标签
- Pull Requests:open/merged、review 情况、合并耗时、变更行数(additions/deletions)
- Commits:提交时间、作者、（可选）增删行数
- Releases:发布时间、tag、发布频率
- Contributors:贡献者与 contributions

关键工程点：
- 分页处理(per_page=100 + page)
- 增量更新(since/updated_at)
- 速率限制处理(rate limit + 退避重试)

### (3) 协作与活跃度指标计算（示例）
- 活跃度：
  - 最近 7/30 天 commit 数、活跃贡献者数
- Issue 响应：
  - 平均首次响应时间（创建→首评）
  - 平均关闭耗时（创建→关闭）
- PR 流程效率：
  - 平均合并耗时(创建→merge)
  - 有 review 的 PR 占比
- 发布节奏：
  - release 次数、平均发布间隔

### (4) 代码热力图与模块热点（可选增强）
- 基于提交文件路径统计“热点文件/热点目录”（修改次数、变更行数）
- 基于 AST/静态分析结果估计复杂度，与热点文件叠加展示
- 输出:Top N 热点模块列表 + 趋势变化

### (5) 健康度评分(Health Score)
- 将多个指标归一化并加权形成 0-100 分
- 输出子分项：活跃度/响应/协作质量/发布节奏
- 支持：仓库对比、时间趋势

# 2.2 代码质量分析子系统(Quality Analysis)
目标：回答“代码是否在变差？主要问题集中在哪些模块？是否出现质量突增？”

### (1) 分析任务管理(Task-based)
- 创建分析任务:选择仓库、分支、工具集(clang-tidy/cppcheck)、配置参数
- 任务状态机:Pending / Running / Finished / Failed
- 定时分析(每周/每次 release 后触发，可选)

### (2) 外部工具调用与结果解析
- 运行 clang-tidy / cppcheck(命令行)
- 解析输出(JSON/XML/文本转结构化)
- 落库:问题实例(文件、行号、规则ID、严重等级、描述、首次出现时间)

### (3) 指标与趋势
- 数量指标:error/warn/style 数量，总问题数
- 密度指标：每千行问题数(可选)
- 趋势：本次 vs 上次新增/修复问题数
- Top 列表：问题最多的文件/目录、最高频规则

### (4) 质量评分（可选）
- 将严重级别加权形成质量分
- 可配置“质量基线”，用于判断是否退化

# 2.3 日志与运行状态监控子系统(Logging & Monitoring)
目标：回答“服务是否稳定？错误是否在爆发？发生故障时能否快速定位？”

### (1) 日志采集方式(推荐:SDK Push)
- 提供 C++ 日志采集 SDK(被监控服务接入)
- 被监控服务通过 SDK 输出结构化日志并异步上报到日志中心
- 可选增强：发送失败重试、本地落盘缓冲

> 说明:GitHub 不能提供第三方项目运行日志；本子系统监控对象为接入的服务/演示服务。

### (2) 结构化日志模型(LogEntry)
每条日志建议包含：
- ts(时间戳)、level(INFO/WARN/ERROR)
- service(服务名)、instance(实例ID)、host/pid/tid(可选)
- msg(文本消息)
- fields/tags(扩展结构化字段，如 errorCode、latencyMs、module 等)
- traceId/spanId(可选,用于关联同一次请求)

### (3) 日志中心服务(Collector)
- 接收接口:HTTP `POST /api/logs/ingest`（批量上报）
- 存储:SQLite/MySQL( SQLite)
- 查询接口：按 service/level/时间范围/关键词过滤 + 分页
- 支持按天滚动存储或基础索引(service、level、ts)

### (4) 指标聚合与告警
- 指标：
  - 每分钟 ERROR 数、总日志数、错误率趋势
  - 按服务/实例的错误排行
- 告警规则（阈值型起步）：
  - 例如:5 分钟内 ERROR > N 触发告警事件
- 告警事件可关联任务(见 #2.4)

# 2.4  AI驱动的智能分析与决策(RAG + 异常检测 + 自动报告 + 团队洞察)
本模块基于 #2.1/#2.2/#2.3 采集与计算出的结构化数据（仓库协作指标、静态分析结果、日志与告警事件），构建可检索的“工程知识库”，并通过 AI/统计方法输出可解释的结论与建议，最终以“AI机器人 + 看板”形式驱动协作闭环。

### (1) 工程知识库(RAG Knowledge Base)
目标：把历史数据沉淀为可检索、可复用的知识，支撑问答与报告生成。

- 知识入库数据源（示例）
  - GitHub:Issue/PR/Commit/Release、Review/Comment 摘要
  - 质量分析:分析任务、规则告警、Top 热点文件、趋势对比结论
  - 日志与告警：告警事件、典型错误日志片段、时间窗统计
  - 项目任务:任务描述、关联证据、处理结论(Done 时的“解决方案总结”)

- 索引与存储（可按阶段实现）
  - 结构化数据:SQLite/MySQL(标准表结构)
  - 文本检索：倒排索引（简化版）或向量索引（可选增强）
  - 文档切分：按 Issue/PR/告警事件/报告段落作为最小 chunk

- RAG 的基本流程
  1) 用户提问/触发报告生成  
  2) 根据 query 从知识库检索 Top-K 相关证据(Issue/PR 片段/告警摘要/指标快照)
  3) 将“证据 + 当前时间窗指标”输入大模型生成回答/报告  
  4) 输出带引用的结果（每条结论附 evidence id/链接，保证可追溯）

### (2) AI问答(工程助手 / AI机器人)
目标：回答“与项目历史数据强相关”的工程问题（强调可引用证据）。

支持问题类型（示例）：
- “这个模块是谁写的/谁最熟？最近主要由谁在改？”
- “上次测试故障（或线上告警）的原因是什么？当时是怎么修的？关联的 PR/任务是什么？”
- “最近质量下降主要集中在哪些文件/规则？应该优先处理什么？”
- “为什么这周 PR 合并变慢？是 review 变少还是积压变多？”

输出要求：
- 回答文本 + 证据引用列表(Issue/PR 链接、告警 id、分析任务 id、关键指标值)
- 若证据不足:明确提示“当前证据不足,需要补充xxx数据/日志”

### (3) 异常检测与风险预警（统计方法 + 规则）
目标：发现“突然变坏/突增”的信号，并产出可行动的预警。

检测对象（示例）：
- 代码变更异常：
  - 单日/单 PR 增删行数异常飙升(churn spike)
  - 热点文件修改次数突然上升
- 协作异常：
  - Issue 首响时间/PR 合并时间突然变长
  - 未处理 issue/PR 积压快速增加
- 质量异常：
  - 严重告警数突然增加
  - 新增问题数明显高于基线
- 运行异常：
  - ERROR 日志数在短时间内突增
  - 告警频率上升

实现思路（可落地、可解释）：
- 基线：移动平均/移动中位数 + 波动范围(MAD/标准差)
- 规则：阈值 + 时间窗(如 5 分钟 ERROR > N)
- 输出：异常点（时间戳/指标值/阈值）+ 影响范围（仓库/模块/服务）+ 建议动作（创建任务/回滚/排查）

### (4) 自动发布周报/项目周报(AI生成)
目标：在每次 Release 或每周定时生成结构化周报，减少人工汇总成本，并形成可追溯档案。

周报内容（建议模板）：
- 本期概览：
  - 新增 commit/PR/Issue 数，主要贡献者
  - 发布版本(tag)、发布时间、变更规模
- 协作与交付：
  - PR 平均合并时间、review 覆盖率、issue 响应
- 质量与稳定性：
  - 静态分析新增/修复问题数、Top 规则
  - 告警次数、错误率趋势、Top 错误摘要
- 风险与建议(AI 生成，带证据引用):
  - 本期最大风险点
  - 下期优先任务建议(可一键生成任务)

输出形式：
- Markdown/HTML 报告（可存档）
- 看板卡片（摘要版）

### (5) 团队洞察:隐形专家识别(PageRank/图算法)
目标：从协作网络中识别“关键贡献者/隐形专家”，辅助任务分派与知识传承。

图建模（示例）：
- 节点：开发者
- 边：
  - A review 了 B 的 PR(A→B)
  - A 关闭/解决了 Issue(A→Issue)
  - A 的 commit 触达热点模块(A→Module)
- 权重：
  - review 频次/合并贡献、修复关键告警、解决高优问题等

输出指标（示例）：
- 隐形专家 Top-K(按模块/全局)
- “谁最擅长处理某类问题”（按规则/错误码/模块聚合）
- 任务推荐负责人（可选，与 #3.2 智能分诊联动）

### (6) 前端看板与交互(Dashboard + AI机器人)
目标：把数据与 AI 输出“可视化、可操作”。

前端页面建议（最小集合）：
- 项目总览 Dashboard:
  - 健康度评分、趋势曲线、风险提示卡片
- 仓库详情页：
  - 协作指标、热点文件/模块
- 质量分析页：
  - 分析任务列表、Top 问题、趋势对比
- 日志与告警页：
  - 日志查询、ERROR 趋势、告警列表与详情
- AI机器人页(或侧边栏):
  - 输入问题 → 返回回答 + 引用证据
  - “一键生成任务/周报”按钮

前端技术实现(可选路线)
- 轻量：静态 HTML + Bootstrap + fetch API + ECharts
- SPA:Vue/React
- 与后端接口：统一走 `/api/...` JSON
